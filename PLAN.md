# VenueOS — ივენთ ჰოლის მართვის სისტემა · სრული გეგმა

**მიზანი:** სრული ბიზნესის მართვა ერთი სისტემით — ჯავშნებიდან ფინანსურ პროგნოზებამდე.
**ობიექტები:** ჯერჯერობით 2, ერთი Postgres ბაზა, ყველა ცხრილი `venue_id`-ით გამიჯნული. ახალი ობიექტის დამატება — ერთი კლიკით.

## სტეკი

| ფენა | ტექნოლოგია | რატომ |
|---|---|---|
| Frontend/Backend | Next.js 16 (App Router) + React 19 + TypeScript | ერთი პროექტი, server actions — API-ს გარეშე |
| სტილი | Tailwind CSS v4 + საკუთარი დიზაინ-სისტემა | სწრაფი, თანმიმდევრული UI |
| ბაზა | **PostgreSQL 14** (ლოკალური) | მრავალობიექტიანი მონაცემები, მყარი ტიპები, მზადაა cloud-ზე გასატანად |
| ORM | Drizzle ORM + drizzle-kit | ტიპიზირებული სქემა, მარტივი მიგრაციები (`npm run db:push`) |
| გრაფიკები | Recharts v3 | პროგნოზები, ფულადი ნაკადი |
| იკონები / ფონტი | lucide-react · Noto Sans Georgian | ქართული ტიპოგრაფია |

პორტი: **4400** (`npm run dev`). ბაზა: `venue_os`.

## არქიტექტურა

- `db/schema.ts` — მთელი სქემა (21 ცხრილი, ქვემოთ)
- `lib/db.ts` — Postgres pool + drizzle
- `lib/venue.ts` — აქტიური ობიექტი cookie-დან; `middleware.ts` → `/select` თუ არჩეული არაა
- `lib/actions.ts` — server actions (მუტაციები), `lib/queries.ts` — კითხვები
- `app/(app)/*` — მოდულების გვერდები საერთო sidebar-ით; `app/select` — ობიექტის ამრჩევი

## მოდულები და ფაზები

### ✅ ფაზა 0 — ჩონჩხი (გაკეთდა)
- პროექტი, Postgres სქემა (ყველა მოდულის ცხრილები წინასწარ), დიზაინ-სისტემა
- ობიექტის ამრჩევი + გადამრთველი sidebar-ში (2 ობიექტი დათესილი)
- **ჯავშნები** — მუშა CRUD: შექმნა (დამკვეთით), სტატუსები, გადახდები, ფილტრები
- **დაფა** — რეალური KPI-ები: მომავალი ივენთები, მოსალოდნელი შემოსავალი, მისაღები თანხა, თვის ბალანსი

### ფაზა 1 — ყოველდღიური ოპერაციები
1. **დღის რეესტრი** — დღის ჟურნალი (შემოსავალი/ხარჯი/ხელფასი), თანამშრომლების ცვლები, დღის დახურვა (Z-რეპორტი, ნაღდის დათვლა, ნაშთის გადატანა), თვის კალენდარი
2. **მომწოდებლები** — ბაზა, შესყიდვები, ვალების კონტროლი

### ფაზა 2 — თვითღირებულება
3. **კალკულაციები** — ინგრედიენტები (დანაკარგის %-ით) → კერძები (food-cost %) → პაკეტები (ღირებულება/სტუმარი ავტომატურად); ფასის ცვლილება ჯაჭვურად ვრცელდება
4. **ინვენტარიზაცია** — მარაგები, მოძრაობა, მინ. ზღვრები, ტევადობის ანალიზი

### ფაზა 3 — ფინანსური სურათი
5. **ფინანსები** — P&L თვეებით, ფიქსირებული ხარჯები, ფულადი ნაკადი, ივენთების მომგებიანობა
6. **პროგნოზები** — 12-თვიანი მოდელი სეზონურობით, what-if სლაიდერები, break-even, სენსიტიურობა, payback

### ფაზა 4 — გაფართოება
- სერვისები (დამატებითი მომსახურება ჯავშანზე), კლიენტების CRM, მობილური ვერსია, cloud deploy (Render/Neon), მომხმარებლები და როლები

## ბაზის სქემა (უკვე შექმნილია Postgres-ში)

- **ბირთვი:** `venues`, `settings` (key/value ობიექტზე)
- **ჯავშნები:** `clients`, `bookings`, `payments`, `services`, `booking_services`
- **რეესტრი:** `staff`, `ledger`, `day_closes`
- **მომწოდებლები:** `suppliers`, `purchases`
- **კალკულაცია:** `ingredients`, `dish_categories`, `dishes`, `dish_ingredients`, `packages`, `package_dishes`
- **ინვენტარი:** `inventory_items`, `inventory_moves`
- **ფინანსები:** `fixed_costs`

ფული — `numeric(12,2)`, სტატუსები — Postgres enum-ები, ყველა მოდულის ცხრილი `venue_id`-ზე cascade-ით.
